# `@subsquid/ponder@0.16.10-sqd.1`

`ponder@0.16.10` + the SQD Portal backfill layer, drop-in (bin stays `ponder`). This is the
first cut on `ponder@0.16.10`. It is **not** the default install — `0.17.0-sqd.1` is the newest
tracked version and holds `--tag latest` — so this cut publishes under the `ponder-0.16.10`
dist-tag; install it by exact version. It carries the same Portal layer as `0.17.0-sqd.1` — the
full invariant-first re-architecture and correctness wave — grafted onto `ponder@0.16.10`.

Install: `npm i @subsquid/ponder@0.16.10-sqd.1` (or `npm i @subsquid/ponder@ponder-0.16.10`).

### Highlights (new since `0.16.8-sqd.1`)

- **Realtime finalize is write-side idempotent on resume — no duplicated factory-child rows
  (INV-17, #175).** A resumed single writer that re-finalizes the same factory-child creation
  block used to insert the `ponder_sync.factory_addresses` row twice (two sequential finalize
  transactions — pre-kill and resume). The read→dedupe guard (keep a child only if it is absent
  or re-discovered at a strictly lower block — LEAST semantics) now runs on the realtime finalize
  path as well as the historical path, so both sync modes are byte-identically idempotent on
  resume. App data was already unaffected (the only consumer min-merges to a set) — this closes
  the store-identity/table-growth defect under repeated realtime crash/resume.
- **`inserted.logs` reports the store-inserted count, not the raw-streamed count (#143, #177).**
  The `inserted.logs` metric (and the completion line's `logs=`) now counts the assembled logs
  actually written to the store, instead of the raw pre-re-match streamed count. A metric-accuracy
  fix — no change to the data that is stored.

### Portal layer

- Write-side idempotence (`portal-child-dedupe.ts`, `runtime/realtime.ts` wiring hook): the
  shared read→dedupe core now runs at BOTH factory-child insert call sites — the historical path
  and the realtime finalize path — each inside its own store transaction (INV-17, #175).
- Metrics (`portal-metrics.ts`, `portal.ts`): `inserted.logs` is taken from
  `assembled.logs.length` at the `insertLogs` site; the raw `logs` count is retained for the
  progress fingerprint (#143, #177).
- Wiring patch: `portal/wiring/0.16.10.patch`. **Byte-identical** to `wiring/0.16.9.patch`
  (a verbatim copy). `0.16.9 → 0.16.10` is a three-file upstream commit-path resilience change —
  `runtime/isolated.ts`, `runtime/multichain.ts`, `runtime/omnichain.ts` only — wrapping the
  per-event checkpoint-update + pglite live-query TRUNCATE in a try/catch that invalidates and
  clears the in-memory indexing cache before rethrowing. Those edits sit ~500 lines below the
  `run*` entry-point region the wiring patch modifies, so they are non-overlapping and the patch
  applies with zero rejects. The graft seam
  (`HistoricalSync.syncBlockRangeData`/`syncBlockData`) is untouched, so the fork passively
  inherits the upstream cache-invalidation resilience fix.

### Fixes

Everything in `0.16.8-sqd.1` (the full correctness wave plus the `accessList` honest-NULL fix
and the default-level Portal logging), plus the two items above: INV-17 realtime-finalize
write-side idempotence (#175) and the `inserted.logs` store-inserted metric (#143, #177). The
fork also passively inherits two off-seam upstream fixes over this range: `ponder@0.16.9`'s
bytea-COPY encoding fix (`ponder-sh/ponder#2323`, `indexing-store/cache.ts`) and
`ponder@0.16.10`'s commit-path cache-invalidation resilience (above) — both off the graft
surface, inherited unchanged.

### Compatibility

  **370/370 across 19 files** on `ponder@0.16.10` (`scripts/sync-upstream.sh 0.16.10 --test`) —

  `0.16.10`, `0.17.0`).

  (`compat.tested`). The seam is stable and identical in shape across `0.15.17 – 0.17.0`.
