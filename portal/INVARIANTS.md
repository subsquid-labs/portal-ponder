# Portal layer — invariant catalog

The `portal/` layer is organised around **explicit, provable invariants**. Every module has a functional
core (pure, unit-tested) behind an imperative shell; the core asserts these invariants at runtime
(`invariant("INV-…", …)` from `portal-invariant.ts`) and the test suite proves them with property-based
tests (`fast-check`, seed-pinned for deterministic CI). IDs are stable (INV-1…INV-25) and grep-able
across **doc ⟷ code ⟷ test**. (INV-19 is the low-edge chunk-coverage guard from #50; INV-20 is the
ancestor-error trace cascade; INV-21 is the root-frame gas parity; INV-22 is the realtime heartbeat
finality cadence; INV-23 is the idle-bounded read + tick-transparent line-wait keeping that cadence
alive on a silent-open connection; INV-24 is the delivery-progress watchdog — a live chain whose /stream
is starving indexing fails loud, a quiet chain never does; INV-25 is the isolated-cutover finality
floor — the isolated single-chain cutover clamp carries a `floor` for parity with the multichain/omnichain
sites, INERT defense-in-depth behind the load-bearing break-before-adopt guard (RG0) — the numbering is
chronological, not table order.)

**Runtime modes** (`PORTAL_CHECKS`, default `on`): `off` disables all checks; `on` runs the O(1) checks
and throws `InvariantViolation` (a loud crash beats silent corruption); `strict` additionally runs the
O(n) whole-structure checks and the assertions that would be too strict against benign upstream retries
(used in CI/tests). Each `InvariantViolation` carries its `id` + a structured `context` and points back
at this file.

| **INV-17** | **Factory-children write-side idempotence.** Re-flushing an already-persisted child set does NOT durably duplicate rows: before every `insertChildAddresses`, the flush is deduped against the store's persisted rows (`getChildAddresses`) and a child is inserted only if it is not yet persisted OR is re-discovered at a STRICTLY LOWER creation block — the write-side analogue of the read-side min-merge (LEAST semantics). ## Invariants in detail

Three invariants — INV-9, INV-10, INV-18 — carry enough machinery that a single table cell is unreadable. Their full statement, rationale, construction-time enforcement, runtime checks, and tests are below, verbatim.

### INV-9 in detail

**Finality safety.** A Portal data request never targets past the known finalized head; an interval ending past the head — or an unknown head — delegates the WHOLE interval to RPC. `dataEnd()` clamps to `min(backfillEnd, head)` (FIX 1), so a BOUNDED backfill whose `toBlock` exceeds the head no longer over-reaches — `desiredTo ≤ head` holds BY CONSTRUCTION whenever the head is known (the old `spec.backfillEnd !== undefined` assert escape is gone; the assert is a strictly-stronger tripwire). An explicit `PORTAL_FINALIZED_HEAD` pin is authoritative for delegation and survives the live-head probe in `ensureChunkSize` (FIX 5): scaling uses the live head, but the pin is NOT clobbered, so an interval above the pin but below the live head still delegates. In stream-realtime mode the RPC fallback is suppressed and BOTH failure shapes are FATAL: an UNKNOWN head at the clamp and delegation seams (the historical↔realtime boundary can't be located, so failing loud beats marking the range synced with no data — fixes **G4**), and — wave 4 — a KNOWN head BELOW the interval (this used to debug + return `[]` as "realtime `/stream` covers it", but the clamp bounds every historical interval at the boundary head and realtime streams only ABOVE it, so a below-interval head here can only be a stale-LOW probe — the `[]` marked the interval synced while NO path ever delivers its data). The head is monotonic upstream; the CACHED head is monotonic too (wave 4): load-balanced replicas answer probes independently, so a later probe can return a lower number, and adopting it regressed the cache into exactly that stale-LOW fatal (or, pre-wave-4, the silent `[]`). The monotonic `max` guarantees only that the cached head is a safe UPPER bound — it never EXCEEDS the true finalized head; it does **not** verify range COMPLETENESS against the specific load-balanced replica that serves a given data POST. **Residual (tracked, #47):** a replica lagging the cached head can 204 a data tail as "above my head", which `dataChunk` records as `coveredTo` — a phantom-coverage hole pre-existing on main (PR #15 FIX 1 documents the 204/truncate shape) that `max`-keeping widens from "until the next lower probe" to the process lifetime. The honest claim is: the cached max never exceeds the true finalized head; completeness against the *serving* replica is not yet verified — see #47 (a client-side range-completeness guard).

**Rationale.** Portal serves only finalized data; its head can lag Ponder's target. Serving the missing tail would 204 and mark it synced (silent hole).

### INV-10 in detail

**Realtime chain linkage.** The unfinalized list is strictly increasing in number and parentHash-linked; `reconcile` yields reorgs whose `reorgedBlocks` are exactly the trimmed suffix; an unknown-parent `gap` is **FATAL** (throws — never silently resets, which would leave already-indexed unfinalized blocks on the wrong fork with no rollback event, fixes **G5**); `finalize` is monotonic and never at/below the startup boundary. The window is ANCHORED at the last finalized block (startup boundary, then each finalize's tip): an EMPTY window appends only a child of the anchor (the anchor itself re-delivered is a duplicate; anything else is a fatal `gap` — the old blind append was undetectable), and a fork whose parent IS the anchor reconciles as a normal reorg off it (was a false-fatal `gap`). A finalize is hash-verifiable only at the probe's EXACT height: a mismatch there is FATAL (never persist a losing fork as finalized), while a hash-carrying finalize whose height is ABOVE the local tip is DEFERRED until the window catches up to a verifiable boundary rather than applied by number on faith (review B1). A `duplicate` explicitly awaited by the consumer (`shouldRedeliver` — the same-block child-discovery redelivery handshake) is re-emitted with the re-delivered batch's logs/txs instead of skipped; it is already the tip, so no append and no linkage check. A finalize that arrives WHILE awaiting a redelivery is HELD in the wire and emitted right after the redelivered block (block N, then finalize N) — not dropped, since `portalRealtimeEvents` has already consumed it (review B2); and the await is watchdog-bounded so a never-redelivering (halted) chain fails loud, not silently stalls. **Fork-negotiation protocol conformance (issue #33): every `/stream` request carries `parentBlockHash` (from a delivered-hash ring seeded with the startup anchor and updated per delivery), including the FIRST (the anchor hash) — the client opts into the Portal's fork detection instead of resuming number-only. A 1-block orphan at tip (an orphaned N−1 then canonical N) is thus HEALED, not fataled: the Portal answers the orphaned resume with HTTP 409 `{previousBlocks}` (the canonical replacement chain), the client rewinds the cursor to the highest `previousBlocks` entry the ring confirms at/above the finalized floor, and the re-served canonical chain flows through `reconcile` as a normal reorg + appends. Fail-safe floors: a missing ring entry at cursor−1 (armed) is FATAL; a fork point at/below the finalized floor is FATAL (no rollback below finality); >10 consecutive 409 rounds WITHOUT cursor progress is FATAL (an OSCILLATION cap, not a per-409 counter — a legitimate no-match step-down descends one height per 409 and is bounded separately by the floor fatal, since `cursor − floor` is finite; only rounds where the cursor did NOT strictly decrease count toward the cap, so the negotiation runs "until a match or the floor" as the docs promise) (F2); the `gap` fatal remains the terminal backstop — no block is ever accepted unlinked. The `gap` and 409-exhausted fatals dump the window/anchor/ring diagnostic (in-memory). AVAILABILITY EDGE (F5, never wrong data): an unfinalized window deeper than RING_CAP (8192) — a fast chain under a max finalize-defer streak — can leave a deep 409 step-down landing on an evicted ring height; the EVICTION fatal ("no delivered-hash ring entry for the resume parent") fires FIRST there, not the floor fatal, because `pruneRing` evicts the LOWEST heights (which sit ABOVE the finalized floor), so the descent crosses into the evicted band before it can reach the floor. The floor fatal fires only when the whole window fits under RING_CAP so nothing is evicted. Either way it is a loud restart, never wrong data (equivalent to today's restart behavior).**

### INV-18 in detail

**Bounded-cutover skip.** A chain whose configured historical end block is at or below its current finalized block (`end !== undefined && end.number <= finalized.number` — "end-capped") is NEVER live-probed at the historical→realtime cutover; and an app in which EVERY chain is end-capped performs ZERO cutover refetches. Unbounded chains (`end === undefined`) and finality-capped chains (`end` still ABOVE finalized) keep today's behavior EXACTLY — they still probe, so the stream-mode finality clamp the wiring applies right after the probe is unchanged for them. The predicate is monotone (finalized only rises across cutover iterations, so an end-capped chain stays end-capped and is never re-added to the probe set). **Refined semantics (adversarial-review findings on the wiring):** (1) **omnichain break requires empty pending** — the all-capped `break` in `getHistoricalEventsOmnichain` fires only when `pendingEvents.length === 0`. The omnichain generator parks events whose checkpoint exceeds the shared omnichain finalized checkpoint (held for cross-chain ordering); an unconditional break would `yield { type: "pending" }` them to realtime, which skips every end-capped chain (`syncProgress.isEnd()`) — silently DROPPING them (bites when one chain's end checkpoint exceeds the min-over-chains finalized checkpoint: a recently-capped end alongside lagging finality elsewhere). Multichain has NO pending mechanism — its break stays plain `shouldSkipCutoverRound`. (1b) **omnichain probe set is NECESSITY-GATED on pending** — the pending-aware break in (1) is necessary but NOT sufficient (second-round review). When every chain is end-capped the probe set `cutoverProbeIndices(...)` is EMPTY, so with `pendingEvents` non-empty the break correctly does not fire — but no chain is probed, so no `finalized` can rise, `omnichainTo` never advances, the parked buffer never drains, and the loop STILL exits via the old `shouldCatchup === false` break with pending non-empty — DROPPING it exactly as (1) described. So the omnichain probe set uses `omnichainCutoverProbeIndices(chains, perChainSync, pendingCount)`: while `pendingCount > 0` it includes EVERY chain (identical to upstream's probe-all), so a refetch raises an end-capped chain's finalized, lifts the min-over-chains finalized checkpoint (hence `omnichainTo`), `isCatchup` flips, and the catchup round drains the buffer; once pending is empty it reduces to `cutoverProbeIndices` (end-capped chains excluded, zero refetches for a fully-bounded app). Because the `pendingCount > 0` path is byte-behaviour-identical to the upstream probe loop (all chains probed + clamped), it inherits upstream's termination guarantee; the reduced (end-capped-excluded) set only applies when `pendingCount === 0` — a pure availability win when there is nothing parked to drop. Phase 1 closes the all-capped + already-parked counterexample (CE-1: an empty probe set can never lift the floor), a convergence hole NEW to this PR (upstream always probes all chains). (1c) **omnichain runs a TWO-PHASE probe — phase 2 re-probes the excluded complement on a catchup round.** Phase 1's necessity gate alone is still NOT sufficient for the MIXED case (CE-2, second-round review): an EXCLUDED end-capped chain can hold a STALE finalized that is the min-over-chains floor, capping `omnichainTo` below an ACTIVE chain's event block, so that event parks on a round where `pendingCount` was still 0 (so the pending gate never re-probes the excluded chain); on a later round the excluded chain advances by ≤ `finalityBlockCount`, `shouldCatchup` stays false, the loop breaks, and realtime drops the parked event. Upstream main never hits this because it probes AND ADOPTS every chain on the catchup round. So `getHistoricalEventsOmnichain` computes `shouldCatchup` from the phase-1 (`omnichainCutoverProbeIndices`) results, and IF — and only if — catchup fires AND chains were excluded, it runs PHASE 2 (`cutoverExcludedIndices`): re-probe + clamp + adopt exactly the excluded complement, so the catchup-round adoption is byte-equivalent to main and no stale finalized can survive as the floor that caps a draining round. A pending-only gate WITHOUT phase 2 is a REJECTED alternative — it still drops CE-2 (proven by the whole-loop tests). When no chain advances (steady-state fully-bounded) `shouldCatchup` is false, phase 2 never runs, and end-capped chains are probed ZERO times (#70's availability win preserved). The remaining "pending non-empty + probe-all + no catchup + every chain realtime-skipped ⇒ realtime drops it" fall-through is identical in upstream main (and the fork's pre-INV-18 clamp-all baseline) — a latent hazard this PR neither introduces nor worsens (out of scope; noted for a follow-up). (2) **excluded chains bypass the clamp** — the post-probe `clampFinalizedToPortalHead` loop skips any chain NOT in `cutoverProbe` (end-capped, excluded from the probe) at BOTH the omnichain and multichain sites. For an excluded chain the clamp is a no-op by construction (its finalized was passed through untouched, floor = same value); in `PORTAL_REALTIME=stream` mode the clamp probes the Portal head and THROWS on an unknown head, which would otherwise drag an excluded stream-mode chain back onto the cutover critical path — defeating the guard. Probed chains keep the clamp exactly as-is (the soaks depend on the stream clamp).

**Rationale.** An end-capped chain's backfill upper bound is `to = min(finalized, end) = end`; the cutover refetch can only RAISE `finalized`, and `min(higher, end) = end` is unchanged — the probe is provably range-neutral (no interval added or removed). Realtime is skipped for such a chain anyway (`syncProgress.isEnd()` ⇒ `getRealtimeEventGenerator` returns early). Yet the probe sits ON the readiness critical path (`/ready` flips only after the generator returns), and a flaky RPC that makes it reject ("Missing or invalid parameters" / `-32602` is retryable per `rpc/index.ts` `shouldRetry` → 10 attempts) delays readiness for minutes and, on retry exhaustion, escapes the fire-and-forget runner as an unhandledRejection that kills a FULLY-INDEXED app with exit 75 before `/ready` ever serves 200. Omnichain ordering's probe is UNCONDITIONAL (no 30s gate), so a permanently-failing RPC blocks completion across restarts.

### Persisted-finality floor (stream-mode restart regression)

The wave-4 monotonic head cache and the wire's `emitFinalize` gate are both IN-MEMORY: neither survives a restart. In stream mode `clampFinalizedToPortalHead` adopts the startup `/finalized-head` probe as the finality boundary, and load-balanced replicas answer independently — so a restart can land on a replica whose head Y sits BELOW finality ponder already PERSISTED (safe checkpoint X, written by `finalizeOmnichain`/`finalizeMultichain` from an earlier finalize). Ponder's own migrate-time guard ("Finalized block for chain … cannot move backwards") checks only the RPC-derived finalized block, which is fetched BEFORE the clamp — the regressed Portal head bypasses it entirely. The consequence is not a rollback (finalized history never forks) but DOUBLE-INDEXING: crash recovery reverts user tables only ABOVE X (rows at/below X lost their reorg-table rows at finalize, by design), historical yields nothing (recovery checkpoint X > boundary Y), and realtime re-streams `(Y, X]` — ponder's realtime event loop has no filter against the recovery checkpoint, so every event in the range executes AGAIN on committed rows. It compounds: the first finalize in `(Y, X]` passes `emitFinalize` (its floor was seeded from the regressed Y) and `finalizeOmnichain` writes the checkpoint VERBATIM, regressing the persisted safe checkpoint itself. The same regression exists mid-run at the backfill-cutover refetch in the omnichain/multichain orderings: `shouldCatchup` is computed per chain but the new boundaries are assigned to ALL chains, so one chain's advance adopts another chain's stale-LOW probe (the isolated ordering is immune — its per-chain `break` refuses a non-advancing boundary).

The fix is the same shape as pipes-sdk's `FinalizedWatermark` (PR #88 there): a monotonic floor SEEDED FROM PERSISTED STATE. `clampFinalizedToPortalHead` takes a `floor` (block number) the boundary never goes below — a head below it is clamped UP to the floor (safe: `(head, floor]` is finalized history, identical across replicas; the lagging replica just hasn't re-marked it final) with a WARN log. The wiring derives that floor from ponder's persisted per-chain safe checkpoint (`crashRecoveryCheckpoint`) via `deriveFinalityFloor` at the startup site (`getLocalSyncProgress`) and passes the previously adopted boundary at the two cutover sites.

`deriveFinalityFloor` distinguishes SAME-CHAIN from FOREIGN checkpoints. In the omnichain ordering `finalizeOmnichain` writes the OMNICHAIN checkpoint verbatim to every chain's row (no per-chain where clause — "it is not an invariant that `chainId` and `checkpoint.chainId` are the same"), so a row's checkpoint can encode ANOTHER chain's block height; used verbatim as a floor that height is wrong in both directions (above the local RPC finalized block it disables the Portal-head clamp outright — the wave-4 stream-mode stale-head FATAL, a deterministic crash-loop; below the local safe point it under-protects). A SAME-CHAIN checkpoint floors at its own encoded block number verbatim (the fast path, `checkpointBlockNumber`). A FOREIGN one — instead of yielding no floor (PR #55's behavior, which left every omnichain chain but one unprotected, issue #57) — is TIMESTAMP-MAPPED: `getSafeCrashRecoveryBlock({chainId: local, timestamp: checkpoint.blockTimestamp})` selects the local chain's highest block with `timestamp < the checkpoint timestamp`, and THAT block is the floor. This mirrors upstream's own crash-recovery handling of a foreign checkpoint verbatim (`runtime/historical.ts` calls the SAME `getSafeCrashRecoveryBlock` seam). Direction-of-error safety (the crash-loop concern): the omnichain checkpoint's `blockTimestamp` is a global finalized watermark, so a local block earlier than it was itself finalized by the time the checkpoint was written — at/below the local finalized height (finalized-block timestamps are monotonic per chain), the SAME safety class as PR #55's same-chain floor (both drawn from prior-run persisted local blocks). The strict `<` never rounds the floor UP past that watermark; and even a floor that somehow exceeded local RPC finality could not push the boundary above it — `clampFinalizedToPortalHead` returns the RPC finalized block unchanged when `boundary >= RPC finalized`, so a too-high floor degrades to the pre-clamp behavior, never a boundary above finality. No local block at/below the timestamp (a first-ever run, or a checkpoint older than every local block) → no floor (pre-#55 behavior, strictly safe).

The floor also overrides a `PORTAL_FINALIZED_HEAD` pin below it — a pin below persisted finality would re-open the double-indexing hole, so correctness wins over the pin. Historical intervals at/below the floor are already cached in the sync store (they were finalized), so the floored boundary triggers no Portal data request a lagging replica would 204. Pinned by `portal-realtime-wire.test.ts` "a probed head BELOW the persisted floor is clamped UP to the floor…", "…floor at/above the RPC finalized block returns it unchanged…", "…the floor overrides a PORTAL_FINALIZED_HEAD pin below it…", the inert-floor case "…a floor at/below the probed head is inert…", the fast-path "…SAME-CHAIN checkpoint uses its own block number directly…", the omnichain-mapping "a FOREIGN checkpoint is TIMESTAMP-MAPPED to the local chain's highest block at/below that timestamp…" and its end-to-end clamp "the timestamp-mapped floor actually CLAMPS the stream-mode boundary UP…", and the unmappable case "…NO local block at/below the timestamp → undefined floor…".
