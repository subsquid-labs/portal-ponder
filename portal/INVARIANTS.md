# Portal layer ‚Äî invariant catalog

The `portal/` layer is organised around **explicit, provable invariants**. Every module has a functional
core (pure, unit-tested) behind an imperative shell; the core asserts these invariants at runtime
(`invariant("INV-‚Ä¶", ‚Ä¶)` from `portal-invariant.ts`) and the test suite proves them with property-based
tests (`fast-check`, seed-pinned for deterministic CI). IDs are stable (INV-1‚Ä¶INV-16) and grep-able
across **doc ‚ü∑ code ‚ü∑ test**.

**Runtime modes** (`PORTAL_CHECKS`, default `on`): `off` disables all checks; `on` runs the O(1) checks
and throws `InvariantViolation` (a loud crash beats silent corruption); `strict` additionally runs the
O(n) whole-structure checks and the assertions that would be too strict against benign upstream retries
(used in CI/tests). Each `InvariantViolation` carries its `id` + a structured `context` and points back
at this file.

### Wave 3 ‚Äî full-repo review (stream-mode realtime)

Two real divergences: a factory child's PRE-CREATION logs (the Portal request carries the child address with no per-address creation floor) and a bounded filter's out-of-range logs (chunks are fetched under the SPEC-global range, so filter B's topics ride along below B's `fromBlock`). `buildEvents` re-filters, so no wrong events fired ‚Äî but the sync store diverged from the RPC path's, breaking the byte-identity claim the validation campaign measures. Now `buildMatchers` exposes `logMatched` (per-filter range + topics + factory creation floor, the exact upstream predicates) and the log branch keeps a log's parent tx/block only for KEPT logs. Mutation-verified: bypassing the re-match fails `portal-assemble.test.ts` "wave 4: a factory child's PRE-CREATION log is excluded‚Ä¶" AND "wave 4: a bounded filter's below-fromBlock log is excluded‚Ä¶".
- **Monotonic head cache + stream-mode stale-head fatal ‚Üí INV-9.** `refreshPortalHead`/`ensureChunkSize` adopted ANY probe result, but load-balanced replicas answer independently ‚Äî a later probe can return a LOWER head, regressing the cache. In stream mode a regressed (stale-LOW) head made an interval at/below the true head read as "past the head", and that branch returned `[]` ‚Äî marked synced while realtime streams only above the boundary: the G4/C11 silent gap, reopened. Two changes: the cached head only ever rises (`max`; every observation is ‚â§ the true head, so the cached max is a safe UPPER bound that never exceeds the true finalized head), and the stream-mode known-head-below-interval branch is now FATAL (it never legitimately fires ‚Äî the clamp bounds historical at the boundary head ‚Äî so reaching it means replica lag persisted through all retries: the same "boundary cannot be located" condition as an unknown head). Mutation-verified: reverting the `max` fails `portal-shell.test.ts` "INV-9: the cached Portal head is MONOTONIC‚Ä¶"; the `[]`‚Üíthrow change is pinned by "INV-9: stream-realtime mode with a KNOWN head below the interval is FATAL‚Ä¶". **Residual (tracked, #47):** the `max`-keep guarantees only that the cached head never OVER-states finality; it does not verify range COMPLETENESS against the specific replica serving a given `/finalized-stream` POST ‚Äî a replica lagging the cached head can 204 a tail that `dataChunk` then records as `coveredTo` (a phantom-coverage hole pre-existing on main ‚Äî PR #15 FIX 1 documents the 204/truncate shape ‚Äî whose exposure `max`-keeping extends from "until the next lower probe" to the process lifetime). The honest claim is: cached max never exceeds the true finalized head; completeness against the serving replica is not yet verified. Fix is a client-side range-completeness guard (#47), deferred pending one empirical check of Portal 204 semantics.
- **Schema-degradation bound keyed per-FIELD, not per-column ‚Üí INV-13 (availability).** `stream()`'s "dropping its field didn't help ‚Üí real error" bound keyed on `err.tag` ‚Äî the BARE column name ‚Äî so a dataset missing the same column in TWO tables (`logs_bloom` in blocks AND transactions: the Monad-style shape FIX 3 exists for; also `nonce`, `gas_used`) dropped the first, then hard-threw on the second table's 400 ‚Üí chunk rejected ‚Üí G1-evicted ‚Üí refetched ‚Üí identical failure: a deterministic crash-loop. Now keyed by the table-qualified `err.fieldKey`; a RE-thrown fieldKey still means its own drop didn't fix its own 400, so the bound survives. Mutation-verified: reverting the keying fails `portal-client.test.ts` "field degradation: the SAME column missing in TWO tables drops each independently‚Ä¶".
- **Stream-mode droppable BLOCK fields ‚Üí INV-10 (availability).** See the B3 completion note above. Mutation-verified: restoring the `tableKey === 'transaction'` restriction fails `portal-realtime.test.ts` "streamHotBlocks: a DROPPABLE BLOCK-field 400 (dataset lacks mix_hash) degrades too‚Ä¶".
- **Bounded realtime `/finalized-head` probe ‚Üí INV-10 (availability).** The wire's `portalFinalizedHead` was a bare `fetch().then(r => r.json())` ‚Äî no timeout, no abort signal, no body bound ‚Äî while its historical twin had been explicitly hardened against exactly this (issue #14 / PR #16). One black-holed connection froze finalize emission mid-run (`portalRealtimeEvents` awaits the probe inline in the block loop) and startup (`clampFinalizedToPortalHead`'s 3-attempt retry never reached attempt 2). The probe implementation is now SHARED: `probeFinalizedHead` in portal-client.ts (two-phase connect-abort + own-the-lock body read, hash-carrying) backs both the client's `finalizedHead()` and the wire's `portalFinalizedHead()` ‚Äî deleting the duplication that let the weak copy exist. Mutation-verified: restoring the bare fetch fails `portal-realtime-wire.test.ts` "portalFinalizedHead: a HUNG probe is BOUNDED‚Ä¶" (times out instead of resolving `undefined`).
