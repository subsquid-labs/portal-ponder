# Portal layer — invariant catalog

The `portal/` layer is organised around **explicit, provable invariants**. Every module has a functional
core (pure, unit-tested) behind an imperative shell; the core asserts these invariants at runtime
(`invariant("INV-…", …)` from `portal-invariant.ts`) and the test suite proves them with property-based
tests (`fast-check`, seed-pinned for deterministic CI). IDs are stable (INV-1…INV-16) and grep-able
across **doc ⟷ code ⟷ test**.

**Runtime modes** (`PORTAL_CHECKS`, default `on`): `off` disables all checks; `on` runs the O(1) checks
and throws `InvariantViolation` (a loud crash beats silent corruption); `strict` additionally runs the
O(n) whole-structure checks and the assertions that would be too strict against benign upstream retries
(used in CI/tests). Each `InvariantViolation` carries its `id` + a structured `context` and points back
at this file.

Legacy `C#` tags from the pre-refactor comments are mapped where they existed.

## The fixed gaps

- **G1 → INV-13.** A rejected chunk promise stayed in `dataCache` forever (poisoned chunk). Now `dataChunk`'s `p.catch` evicts the entry and frees its rows via the per-fetch token. Mutation-verified: reverting the eviction fails `portal-shell.test.ts` "G1: a failed chunk is evicted (rows freed) and a later call refetches…".
- **G2 → INV-3.** `discoveredThrough` advanced before the scan completed; worse, a successor extension that swallowed its predecessor's failure could confirm coverage over an unscanned hole (empirically reproduced in review). Now the confirmed watermark advances only on success, and failures roll back AND propagate through the extension chain. Mutation-verified: re-introducing the swallow fails the interleaving test.
- **G3 → INV-7.** Buffered-row accounting only counted completed chunks; in-flight NDJSON arrays were invisible to backpressure. Now the client's `onRows` hook registers rows per arriving batch against a per-fetch token. Mutation-verified: a no-op hook fails the mid-chunk test.
- **Head-boundary staleness → INV-13** (found in review; fixed on main by PR #5, merged as 802ceed — re-implemented here with behavior parity incl. the `extends` metrics counter; main's #5 regression test passes unchanged). A frontier chunk fetched while the Portal head sat inside its grid range was cached truncated forever; when the head advanced, later intervals in the same chunk were served from the truncated cache and marked synced over a silent gap. Fixed with per-entry `coveredTo` + tail-only EXTEND. One deliberate divergence: on a FAILED extend, main leaves a cached rejection (its pre-existing G1 gap); here the entry is evicted and its rows freed — the G1-consistent recovery this PR introduces.
- **Restart child loss → INV-15** (fixed on main by PR #2; ported into this architecture with all six of its regression tests passing unchanged).
- **Backfill-floor silent gap (C10) → INV-16** (found + fixed by PR #8, @mo4islona; ported here onto `compileFetchSpec`). `backfillStart` took `Math.min` over only the DEFINED `fromBlock`s, so one bounded source (15M) raised the floor past an unbounded source and `chunkRange` clamped its `[0, 15M)` history out — assembled empty and marked synced. Now any undefined `fromBlock` ⇒ floor `0` (symmetric with `backfillEnd`). Mutation-verified: reverting the floor to `Math.min` over the defined-only list fails both the `portal-filters.test.ts` unit assertion (mixed ⇒ 0) and the `portal.test.ts` C10 regression (the unbounded source's block-100 log goes un-inserted).
- **Stream-mode finalized-head probe (C11) → INV-9** (PR #8, @mo4islona). In `PORTAL_REALTIME=stream` the RPC finality-gap fallback is suppressed and the Portal head IS the historical↔realtime boundary. A persistently failing `/finalized-head` probe left the clamp passing the RPC finalized block through (historical then targeted `(portalHead, rpcFinalized]`, which stream mode serves nothing for) and the delegation seam returning `[]` — both marking the range synced while realtime streamed only ABOVE the head we couldn't find: a permanent silent gap. Now BOTH seams fail loud — `clampFinalizedToPortalHead` retries then throws, `portal.ts` throws on an unknown head. Mutation-verified: reverting either throw fails its stream-mode test. NOTE: this changes the pre-existing INV-9 "warn + empty" / clamp "conservative passthrough" contract — deliberately (see INV-9).
- **Unknown-parent realtime gap (C-gap) → INV-10** (PR #8, @mo4islona). A streamed block whose parent is unknown to the unfinalized window (a reorg deeper than the window, or a skipped block) was silently cleared-and-continued, leaving the already-indexed unfinalized blocks on the WRONG fork with no rollback event and dropping the skipped span. Now `portalRealtimeEvents` throws (a rejection escaping the realtime generator hits ponder's `unhandledRejection` → `exit(75)`; a restart re-derives the window cleanly from the finalized head). Mutation-verified: restoring `unfinalized.length = 0` fails "an unknown-parent gap is FATAL". NOTE: changes the pre-existing INV-10 "`gap` resets" contract — deliberately (see INV-10). A walk-parents-and-refetch auto-recovery is a follow-up.

### Wave 2A — triple-blind bug hunt (correctness wave 2A)

Five correctness fixes found by a triple-blind bug hunt (independent lenses); two (BUG-A discovery floor, BUG-B head-clamp) were empirically reproduced. Each fix ships with a mutation-verified regression test — reverting the fix hunk (tests kept) fails the named test, verbatim assertion recorded below.

- **Bounded-backfill head over-reach (BUG-B) → INV-9 / INV-13** (FIX 1). `dataEnd()` was `spec.backfillEnd ?? portalHead ?? ∞`, so with EVERY source bounded (`toBlock` defined) it returned `backfillEnd` and IGNORED the head. A frontier chunk's `desiredTo`/`coveredTo` then extended PAST the head; the Portal serves nothing above its head, so the tail wasn't streamed yet `coveredTo` recorded phantom coverage to `backfillEnd` — and once the head advanced, a later interval in the same chunk blind-hit the stale cache (`desiredTo ≤ coveredTo`) and was marked synced EMPTY: a permanent silent gap on a bounded historical backfill. Now `dataEnd = min(backfillEnd, head)` clamps BOTH, flowing to `desiredTo`/`coveredTo`/`endHint`/`raEnd` and re-arming the INV-13 extend as the head advances; the INV-9 assert's `spec.backfillEnd !== undefined` escape is gone (strictly stronger). Mutation-verified: reverting `dataEnd` to `backfillEnd ?? head ?? ∞` (+ restoring the escape) fails `portal.test.ts` "regression (FIX 1, BUG-B): a BOUNDED backfill whose toBlock is PAST the head is head-clamped…" (`expected undefined not to be undefined` at `expect(b150).toBeDefined()`) AND the discovery-variant test (tail factory child never discovered).
- **Construction-time discovery floor (BUG-A) → INV-3 / INV-15** (FIX 2). - **Account-source receipts (hardening — decouple from upstream's literal-true receipt invariant) → INV-6 / INV-12.** NOT a reachable bug on the pinned versions (0.15.17 / 0.16.6): upstream types `TransactionFilter.hasTransactionReceipt` as the literal `true` and builds every account tx filter with it set (`build/config.ts`), so `needReceipts` was spec-wide true whenever a tx filter existed and receipts could never go sparse relative to tx rows. But that correctness rested on a three-invariant coupling across two codebases (upstream literal-true typing + build-time always-true + the Portal's spec-wide `needReceipts` projection), sitting on an unguarded landmine: ponder's `buildEvents` reads `transactionReceipt.status` on EVERY `type:'transaction'` event to apply `includeReverted`, via a positional cursor over the stored receipts with NO (blockNumber, transactionIndex) identity check, and the RPC path adds every tx-filter-matched tx to `requiredTransactionReceipts` UNCONDITIONALLY (`sync-historical/index.ts`). If upstream ever relaxed account receipts (e.g. for bandwidth), a receipt-less tx row would crash `buildEvents` (`undefined.status`) and a sparse receipt set would silently consult a NEIGHBOR's receipt — dropping successful txs or emitting reverted ones, with the intervals marked synced (permanent). Now the property is locally provable: `txQuery()` always projects `RECEIPT_FIELDS` and the tx-filter assembly branch pushes a receipt for every matched tx — BEFORE its `seenTx` skip, so a tx also matched by a log filter (whose raw row lacks receipt columns unless `needReceipts`) still gets its receipt from the tx query's row; `pushReceipt` dedupes by hash across branches. The tests pin this hardened contract by forcing `hasTransactionReceipt: false` through `any` (a shape upstream's compiler forbids today). Revert-checked: reverting the assembly hunk fails `portal-assemble.test.ts` "a tx-filter-matched tx yields its receipt even without hasTransactionReceipt" + the BOTH-filters companion and `portal.test.ts` "regression: an account source WITHOUT hasTransactionReceipt still inserts a receipt for every matched tx"; reverting the query hunk fails `portal-filters.test.ts` "the tx query ALWAYS projects receipt fields — even without hasTransactionReceipt".

### Wave 3 — full-repo review (stream-mode realtime)

Fixes to the opt-in `PORTAL_REALTIME=stream` path found by the full-repo correctness review. All → INV-10 unless noted.

- **Same-block factory-child log loss (critical).** The `/stream` filter is server-side and snapshotted at connection open, so a child created in block N had its own block-N logs filtered out of the connection N arrived on; the wire widened the filter but `streamHotBlocks` resumed from N+1 — the logs were permanently lost (the interval was marked cached on finalize, surviving restart). Now the REDELIVERY HANDSHAKE: the wire suppresses N's incomplete event and records `awaiting{hash,number}`; `streamHotBlocks` re-opens FROM N; the re-delivered N reconciles as a `duplicate` that `shouldRedeliver` marks awaited and is re-emitted complete — ponder receives exactly ONE event for N, carrying the child's same-block logs. Convergent (the child set only grows). Guards: a different block arriving while awaiting is FATAL (the suppressed block would be silently skipped); a reorg removing the awaited block clears the wait; a finalize covering the awaited block is HELD in the wire and EMITTED right after the redelivered block (ordering: block N, then finalize N) — dropping it lost it, since `portalRealtimeEvents` had already consumed the finalize, so at `endBlock=N`/a halted chain ponder never finalized N (review B2). A REDELIVERY WATCHDOG bounds the await: if N is never re-delivered (a halted chain 204s forever, so no event trips a per-event check), a loud fatal fires instead of a silent stall (recommended).
- **Empty-window blind append / depth-1 boundary fork.** `reconcile` on an empty window (routine after every finalize, and at startup) appended ANYTHING — a skipped block or wrong-fork block at exactly that boundary was undetectable; and a fork whose parent IS the just-finalized block read as a fatal `gap` (the finalized tip had left the window). Now the window is ANCHORED at the last finalized block: empty-window appends require the anchor as parent, and an anchor-parent fork reconciles as a normal reorg off the anchor.
- **Wrong-fork finalize (hash-blind `takeFinalized`).** Finality was applied by NUMBER only; a consumer lagging behind a reorg could persist fork-A blocks as finalized when the Portal finalized fork-B at the same height — permanent wrong data with no rollback. `takeFinalized` is hash-verifiable only at the probe's EXACT height, so: at that height, a hash mismatch is FATAL; ABOVE it (probe references a block above the local tip — no canonical hash for the tip), the finalize is DEFERRED until the window catches up to a hash-verifiable boundary (finality is monotonic; live chains reach it within a poll or two) rather than applied by number on faith (review B1). Number-only behavior is unchanged when the endpoint provides no hash.
- **Reorged-out children never pruned.** Stock `createRealtimeSync` prunes children whose creation block was reorged away (`childAddressesPerBlock`); the stream wire's `applyDiscovered` was add-only, so a reorged-out child kept matching — phantom child events until restart. The wire's reorg branch now deletes children with creation block above the common ancestor (only stream-discovered children can sit there) and narrows the server filter.
- **`PORTAL_FINALIZED_HEAD` pin ignored by the clamp → INV-9.** FIX 5 made the pin authoritative in `portal.ts`, but `clampFinalizedToPortalHead` probed the live head — the two seams disagreed. With pin < live head, the clamp set the boundary at the live head; historical intervals in `(pin, liveHead]` hit portal.ts's "realtime /stream covers it" branch (its head IS the pin) and were marked synced EMPTY while realtime streamed from liveHead+1 — the exact G4/C11 silent gap, reopened. The clamp now honors the pin (no probe when set).
- **Stream events carried no parent transactions.** `toRealtimeSyncEvent` hardcoded `transactions: []`: `event.transaction` (non-optional in the user API) was silently `undefined` on every realtime log event, and the finalize insert stored logs without their tx rows — realtime rows permanently diverging from the historical backfill's (which fetches parent txs). The stream now projects `TX_FIELDS` with `transaction: true` on every log request (the same relation the historical logQuery uses) and threads the txs through to `BlockWithEventData`. The projected parent txs are DEDUPED BY HASH before emit — parity with the historical assembly's `seenTx` set (two matched logs sharing a parent tx, or overlapping log requests, must yield exactly one tx row) (review B4).
- **Deterministic 4xx on `/stream` retried forever (availability).** A non-429 4xx is deterministic; retrying it every second was a silent, permanent tip outage. Now FATAL with an actionable message — UNLESS it is a DROPPABLE-field schema 400 the historical path also degrades (`TX_FIELDS` projects `accessList`, in `DROPPABLE_FIELDS`): the `/stream` reads and parses the 400 body the SAME way `fetchBatch` does (shared `parseSchemaFieldError`), drops the droppable tx field, and retries — so a dataset historical handles fine no longer refuses stream mode (review B3). 429/5xx still retry, and non-OK response bodies are cancelled/read-bounded (no socket leak).
