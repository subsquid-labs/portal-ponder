# Portal layer — invariant catalog

The `portal/` layer is organised around **explicit, provable invariants**. Every module has a functional
core (pure, unit-tested) behind an imperative shell; the core asserts these invariants at runtime
(`invariant("INV-…", …)` from `portal-invariant.ts`) and the test suite proves them with property-based
tests (`fast-check`). IDs are stable (INV-1…INV-14) and grep-able across **doc ⟷ code ⟷ test**.

**Runtime modes** (`PORTAL_CHECKS`, default `on`): `off` disables all checks; `on` runs the O(1) checks
and throws `InvariantViolation` (a loud crash beats silent corruption); `strict` additionally runs the
O(n) whole-structure checks (used in CI/tests). Each `InvariantViolation` carries its `id` + a structured
`context`.

Legacy `C#` tags from the pre-refactor comments are mapped where they existed.

## The three fixed robustness gaps

- **G1 → INV-13.** A rejected chunk promise stayed in `dataCache` forever (poisoned chunk). Now `dataChunk`'s `p.catch` evicts the entry and frees its registered rows, so the next interval retries.
- **G2 → INV-3.** `discoveredThrough` advanced before the scan completed; a failure left an advanced watermark + a rejected promise chain with no recovery. Now `portal-discovery.ts` advances the `confirmed` watermark only on success, rolls back on failure, and replaces the rejected promise.
- **G3 → INV-7.** Buffered-row accounting only counted completed chunks; in-flight NDJSON arrays were invisible to backpressure. Now the client's `onRows` hook registers rows per arriving batch, freed exactly once on eviction/rejection.
